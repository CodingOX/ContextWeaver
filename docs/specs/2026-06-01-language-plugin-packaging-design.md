# ContextWeaver 语言插件命名与分层收口设计

日期：2026-06-01

## 1. 背景

当前仓库同时存在以下三类语言插件：

- 单语言包：`@alistar.max/contextweaver-lang-typescript`、`...-lang-java`、`...-lang-rust` 等
- 聚合包：`@alistar.max/contextweaver-lang-all`
- 历史兼容包：`@alistar.max/contextweaver-lang-ts21`、`@alistar.max/contextweaver-lang-ts22`

其中，`lang-ts21` / `lang-ts22` 实际表达的是 `tree-sitter` 兼容层，而不是 TypeScript 版本或 TypeScript 语言能力范围。这个命名会让用户误以为它们与 `lang-typescript` 属于同一语义维度，进而造成安装路径混乱。

当前代码已经部分体现出正确方向：

- 主包内置仅保留 `JavaScript`、`Python`、`Go`
- `lang-all` 通过组合单语言包实现
- `lang-ts21` / `lang-ts22` 在加载器中已被降级为 legacy 概念，而非默认候选

问题主要集中在对外产品表达，而不是 runtime 机制本身。

## 2. 目标

本次收口的目标是：

- 让新用户无需理解 `tree-sitter` 兼容层，也能知道应该安装哪个包
- 删除历史兼容包，避免继续暴露已经失去主线价值的安装路径
- 统一 README、开发文档、发布文档中的对外叙事
- 让“主包内置 / 默认核心插件 / 按需插件”三层结构清晰可见

## 3. 非目标

本次设计不包含以下内容：

- 不调整主包内置语言集合
- 不重写 runtime 加载机制
- 不引入新的 grammar 支持或新的语言功能
- 不修改搜索、分片、索引行为
- 不在本轮把 `TypeScript` / `Kotlin` / `Java` / `Rust` 直接并入 builtin runtime

## 4. 现状问题

### 4.1 命名语义错位

`lang-ts21` / `lang-ts22` 会被用户自然理解为：

- TypeScript 2.1 / 2.2
- TypeScript 插件的版本差异
- Node 21 / Node 22 适配包

但它们真实表达的是按 `tree-sitter` 主版本兼容层分组的多语言包。命名与真实职责不一致。

### 4.2 暴露维度过多

当前对外同时暴露三套安装心智：

- 按语言安装
- 按兼容层安装
- 按全量安装

对于新用户而言，这三套心智没有必要同时存在。

### 4.3 用户文档仍暴露 legacy 入口

只要主 README 继续出现 `lang-ts21` / `lang-ts22` 的安装命令，新用户就会把它们视为正常推荐入口，而不是兼容包。

### 4.4 内部设计与外部叙事尚未完全一致

代码侧已经把 `lang-ts21` / `lang-ts22` 当作 legacy；但仓库侧仍保留对应包、测试和文档入口，导致“实现已收口，遗留物仍存在”。

## 5. 推荐分层

对外分层固定为三层：

### 5.1 主包内置层

语义：

- 安装主包即可获得的最小可用语言能力

当前范围：

- `JavaScript`
- `Python`
- `Go`

对外文案原则：

- 明确写“主包开箱即用支持这三种语言”
- 不把主包描述成“支持所有语言”

### 5.2 默认核心插件层

语义：

- 主包装上即自动可用，但内部仍保留独立单语言包边界

当前推荐纳入：

- `TypeScript`
- `Kotlin`
- `Java`
- `Rust`

实现原则：

- 不直接并入 builtin runtime
- 继续保留 `lang-typescript`、`lang-kotlin`、`lang-java`、`lang-rust` 单语言包
- 由主包默认依赖并默认加载这些插件，使用户侧体验等同于“核心默认支持”

对外文案原则：

- 明确写“默认核心支持”
- 对用户表现为安装主包后即可使用
- 不要求用户单独理解插件装载细节

### 5.3 按需插件层

语义：

- 非高频语言继续按需安装，避免默认支持边界无限膨胀

当前范围示例：

- `C`
- `C++`
- `C#`
- `PHP`
- `Ruby`
- `Swift`

对外文案原则：

- 明确写“按需扩展”
- 保持单语言安装路径
- 不让这层反向稀释“默认核心支持”的边界

## 6. 命名策略

### 6.1 结论

不建议继续保留 `lang-ts21` / `lang-ts22`。

推荐策略：

- 直接删除兼容包源码、测试、文档和发布入口
- 不再向新用户或维护者暴露这两个过时分组
- 将支持结构完全收敛到 builtin、默认核心插件、按需插件三层

### 6.2 为什么直接删除

保留兼容包的主要价值只剩历史惯性，但它带来的长期成本更高：

- 用户持续看到无意义的历史分组
- 测试矩阵、发布链路、文档说明都会继续膨胀
- “默认核心支持”结构永远会被 legacy 概念干扰

本轮接受 breaking change，换取更清晰的产品边界：

- 删除 `lang-ts21`
- 删除 `lang-ts22`
- 删除对应测试与发布说明
- 在 release note 中明确标注删除原因与替代路径

## 7. 用户安装路径设计

主 README 只保留三种用户路径：

### 7.1 开箱即用用户

文案：

- 主包内置 `JavaScript`、`Python`、`Go`

适用用户：

- 希望安装后立即可用
- 没有额外语言需求

### 7.2 默认核心支持用户

文案：

- 默认核心支持包括 `TypeScript`、`Kotlin`、`Java`、`Rust`

适用用户：

- 希望主流工程语言开箱即用
- 不想单独安装主流语言插件

### 7.3 按需扩展用户

文案：

- 非默认核心语言继续单独安装对应插件

示例：

- `npm install -g @alistar.max/contextweaver-lang-csharp`
- `npm install -g @alistar.max/contextweaver-lang-cpp`
- `npm install -g @alistar.max/contextweaver-lang-swift`

兼容包不再属于任何公开安装路径。

### 7.4 关于 `lang-all`

`lang-all` 不再是推荐主路径。是否保留可作为独立实现决策，但即使保留，也只能定位为补充型安装方式，不能与默认核心支持并列竞争。

## 8. 文档收口规则

### 8.1 README

README 需要明确做到：

- 保留主包内置语言说明
- 保留默认核心支持语言说明
- 保留按需插件安装清单
- 删除 `lang-ts21` / `lang-ts22` 的所有公开安装命令
- 说明 `TypeScript` / `Kotlin` / `Java` / `Rust` 已属于默认核心支持

### 8.2 迁移/发布/开发文档

以下文档需要同步删除兼容包描述或改成历史删除说明：

- 发布清单
- 迁移说明
- 开发者维护文档

但不再把它们当作可安装目标维护。

### 8.3 对外术语统一

统一术语如下：

- `内置语言`
- `默认核心支持`
- `按需插件`

避免继续混用：

- “可选插件” 同时指默认核心与按需插件
- “ts21/ts22 插件” 这类已删除历史概念

## 9. 运行时加载策略要求

本轮不改核心 runtime 机制，但要求文档与实现口径保持一致：

- builtin 继续只承载 `JavaScript`、`Python`、`Go`
- 默认插件候选提升为 `TypeScript`、`Kotlin`、`Java`、`Rust`
- 按需插件继续独立安装、独立加载
- `lang-ts21` / `lang-ts22` 从加载器、测试和发布链中移除

## 10. 迁移策略

迁移原则如下：

### 10.1 对现有用户

- 明确这是 breaking change
- 旧安装路径不再继续维护
- release note 需要给出替代路径：
  - `TypeScript` -> `lang-typescript`
  - `Kotlin` -> `lang-kotlin`
  - `Java` -> `lang-java`
  - `Rust` -> `lang-rust`

### 10.2 对新用户

- 不再看到 `lang-ts21` / `lang-ts22`
- 主包默认具备 `TypeScript` / `Kotlin` / `Java` / `Rust`
- 长尾语言才需要额外安装

### 10.3 对维护者

- 删除兼容包测试与发布步骤
- 更新默认插件候选和安装说明
- 保持默认核心支持边界稳定，不随意继续扩容到所有语言

## 11. 验收标准

完成本设计后的可见结果应满足：

- 新用户只看 README，就能知道：
  - builtin 支持哪些语言
  - 默认核心支持哪些语言
  - 哪些语言仍需按需安装
- 仓库中不再残留 `lang-ts21` / `lang-ts22` 的包、测试、发布步骤
- 文档中的支持分层与运行时实际策略一致

## 12. 推荐实施顺序

建议实施分两步：

1. 先删除兼容包与相关测试/发布链
2. 再提升默认核心插件候选与用户文档说明
3. 最后根据结果决定是否保留 `lang-all`

这样可以先做结构收口，再做文档统一，避免一边保留历史包一边重写叙事。

## 13. 最终结论

当前设计的核心问题已经不是“要不要解释 legacy”，而是“要不要继续保留 legacy”。本轮结论是直接删除，并把产品结构收口为三层：

- builtin：`JavaScript` / `Python` / `Go`
- 默认核心支持：`TypeScript` / `Kotlin` / `Java` / `Rust`
- 按需插件：其余长尾语言

同时，默认核心支持通过“主包装上即自动可用”的方式实现，而不是直接并入 builtin runtime。这样既能给用户简洁的默认体验，也能保留单语言包边界，控制工程风险。
